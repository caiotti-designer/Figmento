# FN-P3: `execute_code` Feasibility Assessment

**Author:** @architect (Aria)
**Date:** 2026-03-25
**Status:** Complete
**Verdict:** FEASIBLE WITH CONDITIONS

---

## Context

The proposal: add a single `execute_code(code: string)` handler to the Figma plugin sandbox, invoked via WebSocket command. The chat AI generates a JavaScript code block (same format as `use_figma` skill examples), sends it over WS in ONE message, and the sandbox executes it directly against the `figma` global. This replaces the need to migrate 35 individual MCP canvas tools to Figma's external `use_figma` API (original Phase 3, blocked on Figma API GA and rate limits).

**Key architectural fact:** The plugin sandbox (`code.ts`) already runs in Figma's QuickJS-based sandbox. There is no `eval`, `Function`, `fetch`, `require`, or `import` available. The only powerful global is `figma`. The code we execute is generated by our own LLM (controlled by our system prompt), not user-typed.

---

## Q1: Security Guardrails

### Threat Model

The Figma plugin sandbox already provides strong isolation:

| Threat | Sandbox Mitigation | Residual Risk |
|--------|-------------------|---------------|
| Filesystem access | No `fs`, `require`, `import` | NONE |
| Network exfiltration | No `fetch`, `XMLHttpRequest`, `WebSocket` | NONE |
| Eval of external code | No dynamic `import()`, no `require` | NONE |
| Cross-file access | `figma.root` scoped to current file | NONE -- cannot open other files |
| Cross-page node access | `figma.getNodeById()` can reach any page in current file | LOW -- LLM unlikely to target other pages, but possible |
| Infinite loops | No built-in timeout in QuickJS sandbox | **HIGH** -- freezes Figma UI thread |
| Mass deletion | `figma.currentPage.children.forEach(c => c.remove())` | **MEDIUM** -- destructive but undo-able (Ctrl+Z) |
| Memory exhaustion | Creating millions of nodes in a loop | **MEDIUM** -- could crash Figma tab |
| Published component mutation | `figma.getNodeById` on component nodes | LOW -- edits stay local, don't affect library |

### MUST-HAVE Guardrails

**1. Execution Timeout (critical)**

The sandbox runs on Figma's UI thread. An infinite loop freezes the entire Figma application -- the user cannot even undo. This is the only truly dangerous scenario.

Implementation: Wrap execution in a `Promise.race` with a timeout. The challenge is that synchronous infinite loops cannot be interrupted by `setTimeout` in the same thread. However, since all Figma Plugin API calls that create or modify nodes are synchronous (not async), we need a different approach:

- **Approach A (recommended):** Inject a loop counter preamble into the code. Before execution, prepend a `__guard__` function that increments a counter on every iteration and throws after N iterations. Rewrite `while`/`for` loops via simple string replacement to include `__guard__()` calls. Max iterations: 10,000 (sufficient for any reasonable design -- 10K node operations is far beyond any real use case).
- **Approach B (simpler, partial):** Wrap the entire execution in `setTimeout(reject, 30000)` and rely on `await` boundaries. This catches async infinite loops but NOT synchronous ones. Since most Plugin API code is synchronous, this alone is insufficient.
- **Recommended:** Use Approach A (loop counter injection) + Approach B (30s hard timeout for async operations) together.

**2. Node Operation Cap**

Cap the number of nodes created per execution to prevent memory exhaustion. Track via a counter incremented on each `figma.create*` call. Limit: 500 nodes per execution (our largest existing design creates ~100 nodes via `batch_execute`).

Implementation: Wrap `figma` in a proxy or inject counter checks after code transformation. Simpler: the loop guard from guardrail #1 already limits total iterations, which implicitly caps node creation.

**3. Scope to Current Page**

Restrict operations to `figma.currentPage`. The LLM-generated code should not navigate to other pages. Block `figma.currentPage = ...` reassignment via static analysis (simple regex: reject code containing `figma.currentPage =` or `figma.setCurrentPageAsync`).

**4. Error Wrapping**

Wrap the entire execution in try/catch so that runtime errors never bubble up to the plugin message handler unhandled. Return structured error information (message, line estimate, partial results).

### NICE-TO-HAVE Guardrails

**5. Pre-Execution Snapshot**

Call `figma.currentPage.children.map(c => c.id)` before execution to capture the set of existing top-level node IDs. After execution (or on failure), we can diff to identify nodes created by the code block -- enabling rollback on request.

**6. API Surface Restriction**

Block access to destructive global operations:
- `figma.closePlugin()` -- would kill the plugin
- `figma.notify()` with extremely long strings -- UI spam
- `figma.clientStorage` -- could corrupt plugin settings

These are low-priority because the LLM is unlikely to generate these calls, and they are non-destructive to user data. Implement via regex blocklist if desired.

**7. Code Size Limit**

Cap `code` string at 50,000 characters (matching `use_figma`'s limit). This prevents absurdly large payloads from consuming memory during parsing.

## Q2: Code Validation Strategy

### Recommended: (d) Hybrid -- lightweight static checks + sandbox trust + timeout

**Reasoning:**

1. **The code is LLM-generated, not user-typed.** The threat model is fundamentally different from a user-facing `eval()`. The LLM is controlled by our system prompt (which we own), and the code format follows the established `use_figma` skill patterns. The LLM will not intentionally generate malicious code.

2. **The Figma sandbox is the real security boundary.** No filesystem, no network, no cross-file access. The sandbox already blocks the most dangerous operations. We are not "trusting" arbitrary code -- we are trusting code that can only call `figma.*` methods on the current file.

3. **AST parsing is overkill.** Adding a JavaScript parser (acorn, babel) to the plugin bundle adds 50-200KB and introduces its own complexity. The code runs in a sandbox that is already more restrictive than anything an AST validator could enforce. The only real threats (infinite loops, mass deletion) are better handled by runtime guardrails (timeout, node cap) than static analysis.

4. **RegExp blocklist catches the few specific patterns we care about.** The blocklist is small and targeted:

```typescript
const BLOCKED_PATTERNS = [
  /figma\.closePlugin/,                    // would kill plugin
  /figma\.setCurrentPageAsync/,            // page navigation
  /while\s*\(\s*true\s*\)/,               // obvious infinite loops
  /for\s*\(\s*;\s*;\s*\)/,                // obvious infinite loops
  /figma\.currentPage\s*=/,               // page reassignment
];
```

This is intentionally bypassable (the LLM could obfuscate) -- but that's fine because (a) the LLM won't try to bypass it, and (b) the runtime loop guard catches anything the regex misses.

### Why NOT full AST validation

- Adds significant bundle size to the plugin (which must stay lean)
- False positives on legitimate code patterns
- Cannot catch runtime behavior (e.g., a loop that becomes infinite based on data)
- The runtime guardrails (loop counter, timeout, node cap) are strictly more reliable

### Why NOT "just trust the sandbox"

Pure option (c) with only a timeout is almost sufficient, but the synchronous infinite loop problem is real. A `while(true){}` in the sandbox freezes Figma with no recovery except force-quit. The loop counter injection (guardrail #1) costs almost nothing and eliminates this risk entirely.

## Q3: Error Reporting

### Error Response Format

```typescript
interface ExecuteCodeResult {
  success: boolean;

  // On success:
  returnValue?: unknown;          // The value returned by the code (if any)
  createdNodeIds: string[];       // All node IDs created during execution
  modifiedNodeIds: string[];      // Node IDs that were modified (best-effort)
  executionTimeMs: number;        // Wall-clock execution time

  // On failure:
  error?: string;                 // Error message
  errorLine?: number;             // Estimated line number (from stack trace parsing)
  errorColumn?: number;           // Estimated column (from stack trace parsing)
  partialNodeIds: string[];       // Nodes created before the error
  rollbackPerformed: boolean;     // Whether auto-rollback was attempted
  codeSnippet?: string;           // The failing line (extracted from input code)
}
```

### Line Number Recovery

JavaScript error stack traces include line numbers, but since we prepend the loop guard preamble, the reported line number will be offset. Strategy:

1. Count the number of lines injected as preamble (the `__guard__` function definition, typically 5-8 lines).
2. When catching an error, parse `error.stack` for line numbers and subtract the preamble offset.
3. Use the adjusted line number to extract the failing line from the original code string.
4. If stack trace parsing fails (QuickJS may not provide detailed stacks), fall back to reporting only the error message.

This is imprecise but sufficient -- the LLM generated the code, so it knows what was on each line. The error message + failing line snippet gives it enough context to fix the issue.

### Partial Results Tracking

Track all created nodes during execution by diffing page children before and after:

```typescript
// Before execution:
const preExistingIds = new Set(
  figma.currentPage.findAll(() => true).map(n => n.id)
);

// After execution (success or failure):
const allIds = figma.currentPage.findAll(() => true).map(n => n.id);
const createdNodeIds = allIds.filter(id => !preExistingIds.has(id));
```

For large canvases, `findAll` is expensive. Optimization: only diff top-level children (`figma.currentPage.children`) and their subtrees if the count changed. Or maintain a running list by proxying `figma.create*` calls (more complex but precise).

**Pragmatic approach:** For V1, capture only top-level children before/after. This misses nodes created deep inside existing frames, but those are the minority case for `execute_code` (which typically creates a new root frame and builds inside it).

### Rollback Strategy

**Recommended: NO automatic rollback. Report partial results and let the caller decide.**

Rationale:
- Figma has built-in undo (`Ctrl+Z`). The user can always undo manually.
- Automatic rollback (deleting created nodes on failure) could delete nodes the user wants to keep (e.g., 29 of 30 elements created successfully before line 30 failed).
- The current `batch_execute` handler does NOT roll back on per-command failures -- it reports success/failure per command and continues. The `execute_code` handler should follow the same philosophy.
- Rollback is complex to implement correctly (need to handle node reparenting, property changes on existing nodes, etc.).

Instead, return `partialNodeIds` so the caller (LLM or MCP server) can decide to:
1. Fix the code and re-run (most common)
2. Delete the partial nodes and retry from scratch
3. Keep the partial results and manually complete

## Implementation Sketch

```typescript
// In command-router.ts, add to the switch:
case 'execute_code': {
  data = await handleExecuteCode(cmd.params);
  break;
}

// New handler: handlers/execute-code.ts
const MAX_ITERATIONS = 10_000;
const MAX_EXEC_MS = 30_000;
const MAX_CODE_LENGTH = 50_000;

const BLOCKED = [
  /figma\.closePlugin/,
  /figma\.setCurrentPageAsync/,
  /figma\.currentPage\s*=/,
];

export async function handleExecuteCode(
  params: Record<string, unknown>
): Promise<Record<string, unknown>> {
  const code = params.code as string;
  if (!code) throw new Error('code string is required');
  if (code.length > MAX_CODE_LENGTH) throw new Error(`Code exceeds ${MAX_CODE_LENGTH} char limit`);

  // Static blocklist
  for (const pat of BLOCKED) {
    if (pat.test(code)) throw new Error(`Blocked pattern: ${pat.source}`);
  }

  // Snapshot top-level children before execution
  const priorIds = new Set(figma.currentPage.children.map(c => c.id));

  // Inject loop guard preamble
  const preamble = `
    let __c__ = 0;
    function __guard__() { if (++__c__ > ${MAX_ITERATIONS}) throw new Error('Loop limit exceeded (${MAX_ITERATIONS} iterations)'); }
  `;
  const guardedCode = preamble + '\n' + code;

  const start = Date.now();
  try {
    // Execute as async IIFE (supports top-level await for font loading)
    const fn = new Function('figma', `return (async () => { ${guardedCode} })();`);
    const result = await Promise.race([
      fn(figma),
      new Promise((_, reject) =>
        setTimeout(() => reject(new Error('Execution timeout')), MAX_EXEC_MS)
      ),
    ]);

    const createdNodeIds = figma.currentPage.children
      .filter(c => !priorIds.has(c.id))
      .map(c => c.id);

    return {
      success: true,
      returnValue: result ?? null,
      createdNodeIds,
      executionTimeMs: Date.now() - start,
    };
  } catch (error) {
    const msg = error instanceof Error ? error.message : String(error);
    const createdNodeIds = figma.currentPage.children
      .filter(c => !priorIds.has(c.id))
      .map(c => c.id);

    // Attempt to extract line number from stack trace
    let errorLine: number | undefined;
    const preambleLines = preamble.split('\n').length;
    const stackMatch = (error as Error)?.stack?.match(/:(\d+):/);
    if (stackMatch) {
      errorLine = parseInt(stackMatch[1], 10) - preambleLines;
      if (errorLine < 1) errorLine = undefined;
    }

    return {
      success: false,
      error: msg,
      errorLine,
      partialNodeIds: createdNodeIds,
      rollbackPerformed: false,
      executionTimeMs: Date.now() - start,
    };
  }
}
```

**Critical implementation note:** `new Function()` may not be available in Figma's QuickJS sandbox. Figma plugins run in a restricted JavaScript environment where `eval` and `Function` constructors are blocked for security. If `new Function` is unavailable, the alternative is to relay the code string to the **plugin UI iframe** (which runs in a standard browser context), execute it there via `eval`/`Function`, and bridge back to the sandbox via `postMessage` for each `figma.*` call. However, this adds massive latency (one `postMessage` round-trip per API call). A second alternative: request that Figma support executes the code block through their internal code execution mechanism (similar to how Dev Mode console works). This is the **primary technical risk** of the proposal and must be validated in a spike.

**UPDATE on `new Function` availability:** The Figma plugin sandbox documentation states that `eval()` and `Function()` are not available in the main sandbox thread. However, the plugin UI iframe (which has standard browser capabilities) CAN use `eval`/`Function`. The feasible architecture is:

1. Sandbox receives `execute_code` command via WS message
2. Sandbox forwards code string to UI iframe via `figma.ui.postMessage`
3. UI iframe wraps code in a `Function` and executes it, BUT it has no access to `figma` global
4. This breaks the entire approach -- the UI iframe cannot call `figma.*` APIs

**Resolution:** The code must be executed in the sandbox context where `figma` is available. Since `new Function` and `eval` are blocked there, the only viable execution strategy is:

- **Option A:** Use the `figma.codegen` API or `figma.widget` runtime -- neither provides arbitrary code execution.
- **Option B:** Use `setTimeout(code_string)` -- some JS engines accept a string argument to setTimeout. This is effectively `eval` and is likely blocked in QuickJS.
- **Option C (recommended):** Pre-parse the code into a sequence of API calls on the MCP server side (not in the plugin), then send them as an enhanced `batch_execute` command with JavaScript expression support. The plugin handler interprets a richer command DSL (variables, conditionals, loops) without needing `eval`.

This changes the architecture significantly. See Risks & Mitigations below.

## Risks & Mitigations

| # | Risk | Impact | Likelihood | Mitigation |
|---|------|--------|-----------|------------|
| 1 | **`new Function`/`eval` blocked in Figma sandbox** | CRITICAL -- entire approach fails if code cannot be executed in sandbox context | HIGH (Figma docs confirm eval is blocked) | Option C: transpile code to enhanced batch commands on MCP server side; or lobby Figma for a `figma.executeCode()` API |
| 2 | Synchronous infinite loop freezes Figma | HIGH -- requires force-quit | MEDIUM (LLM-generated code, but possible) | Loop counter injection (if eval is available) or iteration cap in enhanced batch DSL |
| 3 | Mass node deletion by generated code | MEDIUM -- destructive but undoable | LOW | Pre-snapshot + blocklist for `remove()` on page children |
| 4 | Error line numbers inaccurate in QuickJS | LOW -- debugging inconvenience | HIGH | Report error message + code snippet; LLM can infer location |
| 5 | Large code blocks slow to parse/execute | LOW -- performance degradation | LOW | 50KB limit + 30s timeout |
| 6 | Generated code uses unsupported API methods | MEDIUM -- runtime failure | MEDIUM | System prompt constrains generated code patterns; error reporting enables retry |
| 7 | Plugin bundle size increase | LOW | LOW | Minimal -- handler is ~100 lines, no new dependencies |

## Recommendation

### Verdict: FEASIBLE WITH CONDITIONS

The `execute_code` concept is architecturally sound and solves the right problem (eliminating 35 individual tool round-trips). However, the **primary blocker** is that Figma's plugin sandbox blocks `eval` and `new Function`, which means raw JavaScript strings cannot be executed in the context where `figma` is available.

### Three Viable Paths Forward

**Path A: Enhanced Batch DSL (recommended for V1)**

Instead of executing raw JavaScript, extend `batch_execute` with:
- **Variable declarations** (`let`, references via `$varName`)
- **Conditional execution** (`if` conditions on previous results)
- **Loop constructs** (`repeat N times` with index variable)
- **Expression evaluation** for computed values (arithmetic on dimensions, string concatenation)

This keeps execution in the existing command-router pattern (no `eval` needed), gains most of the single-round-trip benefit, and the LLM generates a structured DSL instead of raw JS. The system prompt already teaches the LLM to generate `batch_execute` commands -- extending the DSL is a smaller prompt change than teaching raw Plugin API JavaScript.

Estimated effort: 2 stories (DSL design + implementation).

**Path B: UI iframe Bridge (technically possible, high latency)**

Forward code to UI iframe for `eval`, proxy every `figma.*` call back to sandbox via `postMessage`. Each API call adds ~5-10ms of message-passing overhead. A 50-operation design goes from ~200ms (direct) to ~500ms (proxied). Usable but not elegant.

Estimated effort: 3 stories (proxy layer + all figma API surface + error handling).

**Path C: Raw `execute_code` (requires Figma platform change)**

If Figma adds a `figma.executeCode(string)` API (analogous to their internal Dev Mode console), this approach works as designed. Monitor Figma's Plugin API changelog. This is the ideal long-term solution but is not actionable today.

### Decision

**Proceed with Path A (Enhanced Batch DSL) for the immediate FN-P3 implementation.** It delivers 80% of the value (single round-trip, variable references, loops) without the `eval` blocker. Keep Path C on the radar as the eventual ideal.

If benchmarking shows Path B's latency is acceptable (~500ms for typical designs), it could serve as an alternative that accepts the exact `use_figma` code format -- making the skill files and system prompts fully portable between Figmento's WS path and Figma's native MCP path.
