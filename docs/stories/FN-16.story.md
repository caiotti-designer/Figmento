## Description

Add a prominent "Use My Design System" toggle to the plugin's generation modes (Chat, Screenshot-to-Layout) that controls whether FN-7 (component matching) and FN-8 (variable binding) are active during design generation. This is the key UX improvement of Phase 2 — it surfaces the design system awareness built in FN-6/7/8 as a single, understandable user control.

### Current State

The plugin's Chat mode (`chat.ts`) and Screenshot mode (`screenshot.ts`) both use `buildSystemPrompt()` from `system-prompt.ts` to construct the AI system prompt. When `designSystemState.cache` is non-null, the system prompt already includes design system context (components, variables, styles). However, there is no user-facing toggle — the DS awareness is always on when a scan has been performed, and always off when it hasn't.

FN-7's `matchComponent()` function (`handlers/component-matcher.ts`) accepts a `cache` parameter and returns null if no cache is provided. FN-8's variable binder (`handlers/variable-binder.ts`) has an `isAutoBindEnabled()` function that reads from `clientStorage` key `figmento-auto-bind-variables` (default: true). These are the two switches this toggle controls.

### How the Toggle Works

- **Toggle ON + DS scanned:** Component matching (FN-7) and variable binding (FN-8) are active. The system prompt includes `designSystemState.cache` context. Generated elements use real components where matched and bind to variables where values match.
- **Toggle ON + DS not scanned:** Toggle is visually disabled with a hint: "Scan your design system first" linking to the DS scan action (in Status Tab per FN-15, or Settings sheet).
- **Toggle OFF:** Behaves like current plugin — primitives only, no DS awareness. The system prompt does NOT include `designSystemState.cache` even if a scan has been performed. `matchComponent()` receives `null` cache. Variable auto-bind is suppressed.

### Placement

The toggle appears in the chat input area (above the message input in Chat mode) and in the screenshot processing step (before the "Start Processing" button). It uses a standard checkbox toggle with a label and is visually prominent — not buried in settings.

## Technical Notes

### System Prompt Integration

`buildSystemPrompt()` in `system-prompt.ts` already accepts an optional `dsCache: DesignSystemCache | null` parameter. When non-null, it appends a "Design System Context" section listing components by category, variables, and styles. The toggle gates whether this parameter is passed as `designSystemState.cache` or `null`.

Current call sites:
- `chat.ts` line (in `sendMessage()`): `buildSystemPrompt(brief, ..., designSystemState.cache)`
- `screenshot.ts` line (in `startProcessing()`): `buildSystemPrompt(undefined, undefined, undefined, designSystemState.cache)`

Both need to be updated to: `buildSystemPrompt(..., isDesignSystemToggleOn() ? designSystemState.cache : null)`

### Component Matching Integration

FN-7's `matchComponent()` is called from the sandbox process (`code.ts` or handlers), not from the UI. The toggle state needs to be communicated to the sandbox via a `postMessage`. When the toggle changes, the UI sends `{ type: 'set-use-design-system', enabled: boolean }` to the sandbox, which stores it and uses it to gate `matchComponent()` calls.

### Variable Binding Integration

FN-8's `isAutoBindEnabled()` already reads from `clientStorage`. The toggle should call `setAutoBindEnabled(enabled)` when its state changes, keeping the two in sync. Alternatively, the toggle can use its own key and the variable binder checks both.

### clientStorage Pattern

The plugin already uses `figma.clientStorage` extensively (see `handlers/storage.ts`). The pattern:
- UI sends `{ type: 'save-setting', key: 'figmento-use-design-system', value: true/false }`
- Sandbox handles with `figma.clientStorage.setAsync(key, value)`
- On init, UI sends `{ type: 'load-setting', key: 'figmento-use-design-system' }`
- Sandbox responds with `{ type: 'setting-loaded', key, value }`

### Toggle UI Pattern

The plugin already has toggle checkboxes (e.g., `#settings-relay-toggle` at `ui.html` line 3676, `#settings-auto-detect` at line 3719). Use the same `.relay-toggle-label` CSS class for visual consistency.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Toggle state desync between UI and sandbox | Medium | High | Use a single source of truth in `clientStorage`; on toggle change, send message to sandbox AND update local state |
| System prompt length increases significantly with DS context | Low | Medium | `buildSystemPrompt()` already handles this; DS context is compact (names + categories, not full definitions) |
| Toggle disabled state confuses users who haven't scanned | Medium | Medium | Clear hint text with actionable scan link; consider auto-scanning on plugin open (out of scope for this story) |
