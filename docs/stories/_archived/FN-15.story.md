## Description

Replace the Bridge connection UI with a unified Status Tab that shows the plugin's awareness of the current Figma file. The current Bridge tab (`#tab-bridge` in `ui.html`, lines 3581-3616) is a WebSocket relay configuration panel — it exposes a URL input, a Connect button, a Channel ID display, an activity log, and command/error counters. This was essential when the plugin's primary interaction model was MCP-server-via-relay, but with Phase 2's design system awareness (FN-6 scanner, FN-7 component matcher, FN-8 variable binder), the primary information a designer needs is what the plugin knows about their current file, not the plumbing of the WS connection.

The Status Tab replaces `#tab-bridge` content with three status cards:

1. **MCP Connection Status** — whether the WS relay is connected, the channel ID (copyable), and command/error counts. This is the same information currently in the Bridge tab but presented as a compact, read-only status card rather than a configuration form. The relay URL input and Connect/Disconnect button move to the Settings sheet (already partially there via `settings-relay-toggle` and `settings-relay-url` at lines 3675-3684 of `ui.html`).

2. **Design System Status** — summary from `designSystemState.cache` (FN-6): component count, variable count, style count, last scanned time, staleness indicator. Currently this information lives only inside the Settings sheet (`#ds-scan-btn`, `#ds-scan-summary`, `#ds-scan-status` at lines 3706-3711). The Status Tab surfaces it prominently with a "Scan" / "Rescan" action button.

3. **Learned Preferences Count** — number of preferences stored in `clientStorage` under the `figmento-preferences` key (from PRD-004 / LC-11). Shows "N preferences learned" with an "Export" link to the Preferences panel in Settings.

### Why This Matters

The Status Tab makes the plugin's intelligence visible. Instead of hiding design system awareness in a settings subpanel, the designer sees at a glance: "I'm connected, my DS has 42 components and 18 variables scanned 5m ago, and the plugin has learned 7 preferences." This builds trust and encourages use of the "Use My Design System" toggle (FN-16).

## Technical Notes

### Current Bridge Tab Structure (ui.html lines 3581-3616)

The Bridge tab contains:
- `#bridge-dot` — status dot (green/gray)
- `#bridge-status` — text: "Connected" / "Disconnected"
- `#bridge-url` — input field, default `ws://localhost:3055`
- `#bridge-connect` — button: "Connect" / "Disconnect"
- `#bridge-channel` — span showing channel ID
- `#channel-hint` — "Click to copy"
- `#bridge-log` — activity log div
- `#bridge-cmd-count` / `#bridge-err-count` — counters

### Bridge State (bridge.ts)

`bridge.ts` exports: `initBridge()`, `autoConnectBridge()`, `getBridgeChannelId()`, `getBridgeConnected()`, `sendBridgeMessage()`, `handleBridgeCommandResult()`, `setClaudeCodeResultHandler()`.

Internal state: `ws`, `bridgeChannelId`, `isBridgeConnected`, `bridgeCommandCount`, `bridgeErrorCount`.

The `initBridge()` function binds a click listener to `#bridge-connect`. This must be updated to the relocated element ID.

### Design System State (state.ts + index.ts)

`designSystemState` in `state.ts` (line 347): `{ cache: DesignSystemCache | null, isScanning: boolean }`.

`handleDesignSystemScanned()` in `index.ts` (line 55) updates the cache and sets DOM text on `#ds-scan-summary` and `#ds-scan-status`. These elements are currently in the Settings sheet; the Status Tab will have its own summary elements that also get updated.

### Preferences Storage (handlers/storage.ts)

Preferences are stored in `clientStorage` under key `figmento-preferences` (defined as `PREFERENCES_STORAGE_KEY` in `handlers/storage.ts`). The plugin already has a `get-preferences` message type. The Status Tab should send this message on init and listen for the response to get the count.

### Tab System (index.ts)

`initUnifiedTabs()` (line 426) uses `.unified-tab-btn[data-tab]` buttons and `.unified-tab-content` divs. The Status Tab will use this same system — the `data-tab` value changes from `tab-bridge` to `tab-status`.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Relocating Bridge DOM elements breaks `bridge.ts` event listeners | Medium | High | Keep the same element IDs where possible; update `initBridge()` to query by new IDs only for relocated elements |
| Status Tab cards don't update reactively when state changes | Medium | Medium | Use the existing callback pattern (e.g., `handleDesignSystemScanned` already updates DOM); add similar callbacks for bridge state and preferences count |
| Preferences count requires an async `clientStorage` read on tab switch | Low | Low | Cache the count in UI state; update on preference changes via existing message handlers |
